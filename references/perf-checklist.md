# Performance Gate — 스킬 실행 성능 체크리스트

**Purpose:** skill-builder가 스킬을 만들거나 수정할 때, 그 스킬이 **실행될 때의 성능**을 반드시 고려하게 하는 게이트.

## 왜 필요한가

- LLM 스킬은 토큰=비용=시간. 불필요한 텍스트 지시가 코드로 대체 가능하면 90% 토큰 절감
- 허브스포크 미적용 → 매 실행마다 전체 SKILL.md 로딩 → 불필요한 컨텍스트 소비
- 스킬 내 "LLM이 텍스트로 판단"하는 절차 중 코드로 대체 가능한 것이 병목

## 실행 속도 저하 7대 원인

스킬이 발동→실행될 때 체감 속도를 떨어뜨리는 구조적 원인. 최적화 판단의 진단 프레임으로 사용.

| # | 원인 | 증상 | 대응 | 기존 체크항목 매핑 |
|---|------|------|------|-------------------|
| 1 | 파일 I/O 횟수 | Read 호출 3회+ 연쇄, 보조파일 순차 로딩 | 보조파일 통합, 조건부 로딩(`→ references/` 포인터) | #1 허브스포크, #2 불필요로딩 |
| 2 | 스킬 프롬프트 길이 | SKILL.md 10KB+, 토큰 30K+ | Lean 원칙, 허브스포크 분리 | #4 토큰예산 |
| 3 | 다단계 스킬 체이닝 | 스킬 A→B→C 순차 호출, 각 SKILL.md 재로딩 | cascade 깊이 제한, 필수 경로만 체이닝 | #5 병목구간 |
| 4 | 서브에이전트 호출 | Agent tool로 새 프로세스 생성 + 내부 tool call 누적 | 에이전트 최소화, 본체 처리 우선 | (신규) |
| 5 | 외부 도구 호출 | pip install, MCP 네트워크, 무거운 Bash 실행 | 사전 설치 전제, 호출 횟수 최소화 | (신규) |
| 6 | 직렬 실행 구조 | 독립 작업을 순차 처리, 병렬 가능한 Read/Edit 직렬화 | 병렬 tool call 적극 활용 | (신규) |
| 7 | QC/검증 루프 | 만들기→검증→수정 2~3회전 반복 | 루프 하드캡(max 2), 1회 통과 목표 설계 | #3 코드대체 |

## 속도 vs 품질 트레이드오프 금지선

> 속도 최적화의 상한선은 "품질·안전성 동등 유지". 더 빠르되 더 나쁜 산출물은 최적화가 아니라 퇴화다.

| 금지 행위 | 이유 | 올바른 대안 |
|-----------|------|------------|
| QC/검증 단계 삭제 | 품질 붕괴. 오류가 사용자에게 그대로 전달됨 | 검증을 코드 자동화(scripts/)로 빠르게 수행 |
| 필수 참조파일 로딩 생략 | 도메인 지식 누락 → 산출물 품질 저하 | 조건부 로딩으로 "필요할 때만" 로딩. 필요한 건 반드시 로딩 |
| 에러 핸들링/재시도 제거 | 실패 시 복구 불가, 사용자에게 깨진 결과 전달 | 재시도 횟수 캡(max 2)으로 제한하되 제거하지 않음 |
| 사용자 컨펌 단계 스킵 | L3/L4 수정이 무확인 실행 → 안전성 훼손 | EDIT4 게이트 준수. 속도를 위해 게이트 레벨을 낮추지 않음 |
| Gotchas/예외처리 축소 | 실패 패턴 대응 누락 → 동일 실패 반복 | Gotchas는 표 형태로 압축하되 항목 자체는 보존 |
| 예시 전면 삭제 | LLM 출력 품질 급락. 예시 없으면 자의적 해석 | 핵심 예시 1개는 반드시 유지. 다수일 때 대표 1개로 축소 |

## 적용 시점

편집 단계(②) 완료 직전, 검증 체크리스트와 함께 1회 실행

## 체크리스트 (5항목)

| # | 체크 | 질문 | 자동화 방법 | FAIL 조건 |
|---|------|------|------------|----------|
| 1 | 허브스포크 적극 활용 | → SKILL.md §허브스포크 판정 참조. validate.py → hub_spoke 필드로 자동 판정 | `python scripts/validate.py` → hub_spoke 필드 | hub_spoke="recommended" 이면서 미조치 |
| 2 | 불필요 로딩 방지 | @uses의 모든 참조가 실제로 매 실행마다 필요한가? 조건부 로딩(`→ references/x.md 참조`)과 구분 | 본문에서 `→ references/` 포인터 vs @uses 비교 | @uses에 있지만 특정 모드에서만 쓰이는 참조 → 조건부로 전환 |
| 3 | 코드 대체 가능성 | 검증·측정·판정 중 grep/wc/python 한 줄이면 되는 것을 LLM 텍스트 추론으로 처리하고 있지 않은가? | 체크리스트 패턴(- [ ]) 중 코드화 가능 항목 식별 | 코드화 가능한 체크가 3개+ 남아있으면 scripts/ 신설 권고 |
| 4 | 토큰 예산 | SKILL.md + @uses references 합계 토큰이 과다하지 않은가? | `python scripts/validate.py` → combined_tokens_estimate | 합계 >30K 토큰이면 경고, >50K면 FAIL |
| 5 | 병목 구간 식별 | 스킬 실행 시 가장 많은 턴/토큰을 소비하는 Phase는? N개 Phase 중 1개가 50%+ 비중이면 분리 또는 코드화 대상 | Phase/Step 수 대비 각 Phase의 텍스트 비중 측정 | 단일 Phase가 전체의 50%+ → 해당 Phase 분리 또는 스크립트화 제안 |

## 코드 vs 텍스트 판별 기준

| 작업 유형 | 코드(scripts/) | 텍스트(SKILL.md) |
|-----------|---------------|-----------------|
| 크기 측정 (bytes, lines, words) | ✓ | |
| 패턴 존재 확인 (grep) | ✓ | |
| YAML 파싱·검증 | ✓ | |
| 수치 비교 (임계치 판정) | ✓ | |
| 구조 분석 (섹션 수, 깊이) | ✓ | |
| 의미 판단 (품질, 적합성) | | ✓ |
| 맥락 의존 판단 (도메인, 사용자) | | ✓ |
| 창의적 생성 (설계안, 대안) | | ✓ |

## Gotchas

- **코드화 과잉:** 의미 판단까지 코드로 하려 하면 false positive 폭발. 코드는 "측정"에만
- **토큰 예산 근사치:** `word_count × 1.3`은 영어 기준. 한국어는 ×2.0이 더 정확하지만, 혼합 문서는 1.3~1.5 사이
- **허브스포크 전환 시:** 기존 @uses 재검토 필수. 분리 후 포인터가 깨지지 않았는지 확인
