# Performance Gate — 스킬 실행 성능 체크리스트

**Purpose:** skill-builder가 스킬을 만들거나 수정할 때, 그 스킬이 **실행될 때의 성능**을 반드시 고려하게 하는 게이트.

## 왜 필요한가

- LLM 스킬은 토큰=비용=시간. 불필요한 텍스트 지시가 코드로 대체 가능하면 90% 토큰 절감
- 허브스포크 미적용 → 매 실행마다 전체 SKILL.md 로딩 → 불필요한 컨텍스트 소비
- 스킬 내 "LLM이 텍스트로 판단"하는 절차 중 코드로 대체 가능한 것이 병목

## 적용 시점

편집 단계(②) 완료 직전, 검증 체크리스트와 함께 1회 실행

## 체크리스트 (5항목)

| # | 체크 | 질문 | 자동화 방법 | FAIL 조건 |
|---|------|------|------------|----------|
| 1 | 허브스포크 적극 활용 | SKILL.md >5KB인데 references/ 없으면 분리 필요 | `python scripts/validate.py` → hub_spoke 필드 | hub_spoke="recommended" 이면서 미조치 |
| 2 | 불필요 로딩 방지 | @uses의 모든 참조가 실제로 매 실행마다 필요한가? 조건부 로딩(`→ references/x.md 참조`)과 구분 | 본문에서 `→ references/` 포인터 vs @uses 비교 | @uses에 있지만 특정 모드에서만 쓰이는 참조 → 조건부로 전환 |
| 3 | 코드 대체 가능성 | 검증·측정·판정 중 grep/wc/python 한 줄이면 되는 것을 LLM 텍스트 추론으로 처리하고 있지 않은가? | 체크리스트 패턴(- [ ]) 중 코드화 가능 항목 식별 | 코드화 가능한 체크가 3개+ 남아있으면 scripts/ 신설 권고 |
| 4 | 토큰 예산 | SKILL.md + @uses references 합계 토큰이 과다하지 않은가? | `python scripts/validate.py` → combined_tokens_estimate | 합계 >30K 토큰이면 경고, >50K면 FAIL |
| 5 | 병목 구간 식별 | 스킬 실행 시 가장 많은 턴/토큰을 소비하는 Phase는? N개 Phase 중 1개가 50%+ 비중이면 분리 또는 코드화 대상 | Phase/Step 수 대비 각 Phase의 텍스트 비중 측정 | 단일 Phase가 전체의 50%+ → 해당 Phase 분리 또는 스크립트화 제안 |

## 코드 vs 텍스트 판별 기준

| 작업 유형 | 코드(scripts/) | 텍스트(SKILL.md) |
|-----------|---------------|-----------------|
| 크기 측정 (bytes, lines, words) | ✓ | |
| 패턴 존재 확인 (grep) | ✓ | |
| YAML 파싱·검증 | ✓ | |
| 수치 비교 (임계치 판정) | ✓ | |
| 구조 분석 (섹션 수, 깊이) | ✓ | |
| 의미 판단 (품질, 적합성) | | ✓ |
| 맥락 의존 판단 (도메인, 사용자) | | ✓ |
| 창의적 생성 (설계안, 대안) | | ✓ |

## Gotchas

- **코드화 과잉:** 의미 판단까지 코드로 하려 하면 false positive 폭발. 코드는 "측정"에만
- **토큰 예산 근사치:** `word_count × 1.3`은 영어 기준. 한국어는 ×2.0이 더 정확하지만, 혼합 문서는 1.3~1.5 사이
- **허브스포크 전환 시:** 기존 @uses 재검토 필수. 분리 후 포인터가 깨지지 않았는지 확인
